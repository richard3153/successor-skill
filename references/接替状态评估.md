# 接替状态评估参考

## 状态评估等级

### 🟢 完整接替（Green）
- 所有上下文文件存在且最新
- 任务状态清晰，无悬空操作
- 用户偏好已记录
- 所有项目文件完整
→ 可直接进入工作状态

### 🟡 部分接替（Yellow）
- 部分文件缺失或过时
- 有进行中的任务但状态不明
- 部分项目文件不完整
- 需要用户确认某些信息
→ 先澄清，再工作

### 🔴 断档接替（Red）
- 关键文件全部缺失
- 没有任何历史记录
- 无法确定用户身份或任务
- 工作区无任何项目文件
→ 主动询问用户，建立基础上下文

## 常见断档场景

| 场景 | 风险 | 处理方式 |
|------|------|----------|
| 新智能体首次启动 | 高 | 读取 BOOTSTRAP.md（如有），主动询问用户 |
| 长时间断档后恢复 | 中 | 搜索 memory 文件，找最近状态，扫描文件 |
| 用户换了设备/渠道 | 低 | 检查 USER.md 是否匹配，扫描工作区 |
| 任务中途接替 | 高 | 先读取 memory，再确认任务目标，扫描项目 |
| 项目文件被移动/删除 | 高 | 询问用户文件位置，在回收站查找 |

## 记忆优先级

1. **必须记录**：任务决策、截止时间、用户偏好变更、所有项目位置
2. **应当记录**：工作进展、遇到的问题、解决方案、文件状态
3. **可以记录**：有趣的上下文、非工作相关的用户信息

## 文件完整性检查

对于任何正在进行的项目，接管时必须检查：

- [ ] 项目根目录的 README.md
- [ ] 依赖配置文件（requirements.txt, package.json 等）
- [ ] 核心代码/数据文件
- [ ] 最近修改的文件（可能是当前工作）
- [ ] .git 状态（如为 Git 仓库）
- [ ] 环境配置文件（.env, config.*）
